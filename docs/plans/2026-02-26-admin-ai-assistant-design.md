# 后台管理系统AI小助手 - 产品设计文档

## 一、产品概述

### 1.1 产品定位
为复杂的后台管理系统提供智能AI助手，帮助用户快速理解系统功能、查询数据和执行操作。类似腾讯云、阿里云等平台的AI助手功能，但专注于内部使用场景。

### 1.2 核心价值
- 降低复杂后台系统的学习成本
- 提供智能的系统指引和页面导航
- 简化常用数据查询操作
- 支持自然语言交互的快捷操作

### 1.3 使用场景
- 内部团队使用
- 可作为开源项目或交付项目供他人二次开发
- 不考虑多租户SaaS架构

## 二、系统架构

### 2.1 整体架构
系统由三个主要部分组成：

**1. AI助手前端组件（admin-ai-widget）**
- 用原生JavaScript开发的独立组件
- 提供聊天界面UI（可展开/收起的悬浮窗）
- 通过script标签引入，支持任何前端框架集成
- 与宿主系统通过自定义事件（CustomEvent）通信

**2. 助手配置后台（admin-ai-console）**
- 独立的后台管理系统
- 功能模块：
  - AI模型配置（API密钥、模型选择）
  - 知识库管理（Markdown文档的在线编辑和管理）
  - 路由配置（页面路径和描述的维护）
  - 数据查询模板配置（预定义查询接口）
  - 快捷操作模板配置（预定义操作接口）

**3. AI助手后端服务（admin-ai-service）**
- 提供统一的API接口
- AI模型适配层（统一不同AI服务商的接口）
- 知识库检索和RAG（检索增强生成）
- 对话历史管理

### 2.2 通信机制
前端组件与宿主系统通过浏览器原生CustomEvent进行通信，保证技术栈无关性。

**示例1：页面跳转**
```javascript
// 组件发出导航事件
window.dispatchEvent(new CustomEvent('ai-assistant-navigate', {
  detail: { path: '/user/add' }
}))

// 宿主系统监听并处理
window.addEventListener('ai-assistant-navigate', (e) => {
  router.push(e.detail.path)
})
```

**示例2：数据查询**
```javascript
// 组件请求数据
window.dispatchEvent(new CustomEvent('ai-assistant-query', {
  detail: { queryId: 'daily-new-users' }
}))

// 宿主系统返回数据
window.dispatchEvent(new CustomEvent('ai-assistant-query-result', {
  detail: { queryId: 'daily-new-users', data: { count: 25 } }
}))
```

## 三、核心功能设计

### 3.1 功能优先级
1. **第一优先级**：系统解读和使用指引（包含页面跳转）
2. **第二优先级**：数据查询和展示
3. **第三优先级**：快捷操作

### 3.2 系统解读和使用指引

**知识库管理：**
- 支持Markdown格式文档
- 在线编辑器进行创建和编辑
- 文档分类管理（功能指引、常见问题、API文档等）
- 每个文档包含：标题、内容、关联的页面路由
- 后端将文档内容向量化存储，用于语义检索

**路由配置：**
维护页面路由表，每条记录包含：
- 路由路径（如 `/user/list`）
- 页面名称（如 "用户列表"）
- 页面描述（如 "查看和管理所有用户"）

**对话流程：**
1. 用户输入问题："如何查看用户列表？"
2. 前端发送到后端服务
3. 后端通过RAG检索相关知识库文档和路由信息
4. 调用AI模型生成回答，包含Markdown格式的链接
5. 前端渲染回答，链接点击时触发`ai-assistant-navigate`事件
6. 宿主系统监听事件并执行路由跳转

**回答示例：**
```
您可以在[用户管理 > 用户列表](#/user/list)页面查看所有用户。
该页面支持搜索、筛选和批量操作。
```

### 3.3 数据查询和展示

**查询模板配置：**
在配置后台预定义数据查询模板，每个模板包含：
- 模板ID（如 `daily-new-users`）
- 查询名称（如 "今日新增用户数"）
- 查询描述（帮助AI理解何时使用此查询）
- API路径（如 `/api/statistics/users/daily`）
- 参数定义（如日期范围、状态等）
- 返回数据格式说明

**对话流程：**
1. 用户提问："今天新增了多少用户？"
2. 后端AI理解意图，匹配到查询模板 `daily-new-users`
3. 前端组件触发查询事件，携带查询ID和参数
4. 宿主系统监听事件，调用自己的API获取数据
5. 宿主系统通过事件返回数据
6. AI助手格式化展示："今天新增了25个用户，比昨天增长15%"

**数据展示方式：**
- 文字描述
- 简单的表格
- 基础图表（可选，后期扩展）

### 3.4 快捷操作

**操作模板配置：**
在配置后台预定义快捷操作模板，每个模板包含：
- 操作ID（如 `create-user`）
- 操作名称（如 "创建用户"）
- 操作描述（帮助AI理解何时使用）
- API路径和方法（如 `POST /api/users`）
- 必填参数和可选参数定义
- 确认提示文案

**对话流程：**
1. 用户说："帮我创建一个用户，用户名是张三"
2. AI理解意图，匹配到操作模板 `create-user`
3. AI识别出已提供参数（用户名：张三），缺少必填参数
4. AI反问："好的，请提供该用户的手机号和邮箱"
5. 用户补充信息后，前端组件显示确认弹窗
6. 用户确认后，触发操作事件
7. 宿主系统执行操作并返回结果
8. AI助手显示操作结果

**安全考虑：**
- 所有操作都需要用户明确确认
- 敏感操作（删除、修改权限等）需要二次确认

## 四、技术选型

### 4.1 前端组件（admin-ai-widget）
- **技术栈**：原生JavaScript + Web Components
- **样式**：CSS（支持主题定制）
- **打包工具**：Rollup或Vite（输出UMD格式）
- **Markdown渲染**：marked.js
- **使用方式**：
```html
<script src="admin-ai-widget.js"></script>
<script>
  new AdminAIWidget({
    apiUrl: 'https://your-api.com',
    position: 'bottom-right'
  })
</script>
```

### 4.2 配置后台（admin-ai-console）
- **前端框架**：Vue 3 + Element Plus（或React + Ant Design）
- **Markdown编辑器**：集成现成的编辑器组件
- **路由**：Vue Router / React Router

### 4.3 后端服务（admin-ai-service）
- **框架**：Node.js + Express（或Nest.js）
- **数据库**：
  - MySQL：存储配置数据
  - 向量数据库：Milvus或PostgreSQL + pgvector（存储知识库向量）
- **AI模型适配**：
  - OpenAI（ChatGPT）
  - 通义千问
  - 文心一言
  - 智谱AI
  - 统一适配层处理不同API格式

### 4.4 部署方案
- **前端组件**：CDN或静态文件服务
- **配置后台**：独立部署
- **后端服务**：Docker容器化部署

## 五、数据流设计

### 5.1 完整对话流程

**1. 初始化阶段：**
- 宿主系统引入AI助手组件脚本
- 组件初始化时传入配置（API地址、项目ID等）
- 组件从后端获取基础配置（主题、欢迎语等）

**2. 用户提问：**
- 用户在组件中输入问题
- 组件发送请求到后端服务：`POST /api/chat`
- 携带：用户问题、对话历史、项目ID

**3. 后端处理：**
- 接收问题，进行意图识别
- 从向量数据库检索相关知识库内容
- 匹配路由配置、查询模板或操作模板
- 调用AI模型生成回答（携带检索到的上下文）
- 返回结构化响应：
```json
{
  "type": "text|query|action",
  "content": "AI回答内容",
  "actions": [
    {
      "type": "navigate|query|execute",
      "data": {...}
    }
  ]
}
```

**4. 前端渲染和交互：**
- 渲染AI回答
- 如果包含链接，点击时触发导航事件
- 如果需要查询数据，触发查询事件并等待宿主系统返回
- 如果需要执行操作，显示确认弹窗

**5. 对话历史：**
- 后端存储对话历史（可选择保留时长）
- 支持上下文理解（"刚才那个页面在哪"）

## 六、错误处理和边界情况

### 6.1 AI理解错误
- 当AI无法理解用户意图时，回复："抱歉，我没有理解您的问题，您可以换个方式问问看"
- 提供常见问题快捷入口

### 6.2 知识库未覆盖
- 当知识库中没有相关内容时，AI诚实回答："抱歉，我暂时没有找到相关信息"
- 提供反馈入口，记录未覆盖的问题供后续完善

### 6.3 API调用失败
- 查询或操作失败时，显示友好的错误提示
- 记录错误日志供排查

### 6.4 网络异常
- 组件显示加载状态
- 超时后提示用户重试
- 支持离线提示

### 6.5 数据安全
- 敏感信息（API密钥）加密存储
- 对话历史可配置保留时长或不保留
- 操作日志记录（谁在什么时间执行了什么操作）

### 6.6 降级方案
- 当AI服务不可用时，组件显示"助手暂时不可用"
- 提供静态的帮助文档链接作为备选

## 七、开发计划

### 7.1 第一阶段 - MVP（最小可行产品）
**目标**：实现系统解读和使用指引功能

**前端组件：**
- 聊天界面UI（展开/收起）
- 基本对话功能
- 支持Markdown渲染和链接点击

**配置后台：**
- AI模型配置（先支持1-2个模型）
- 知识库管理（Markdown在线编辑）
- 路由配置管理

**后端服务：**
- 对话API
- 简单的RAG检索（可以先不用向量数据库，用关键词匹配）
- AI模型适配层

### 7.2 第二阶段 - 数据查询
**目标**：实现数据查询和展示功能

- 配置后台增加查询模板管理
- 前端组件支持事件通信机制
- 后端增加查询模板匹配逻辑
- 数据格式化展示

### 7.3 第三阶段 - 快捷操作
**目标**：实现快捷操作功能

- 配置后台增加操作模板管理
- 前端组件增加确认弹窗
- 后端增加操作模板匹配和参数收集逻辑

### 7.4 第四阶段 - 优化增强
**目标**：提升用户体验和系统能力

- 引入向量数据库提升检索准确度
- 增加对话历史和上下文理解
- 支持更多AI模型
- 权限控制
- 数据可视化（图表）

## 八、非功能性需求

### 8.1 性能要求
- AI响应时间：< 3秒
- 组件加载时间：< 1秒
- 支持并发对话数：根据实际需求确定

### 8.2 兼容性
- 浏览器：Chrome、Firefox、Safari、Edge（最新两个版本）
- 移动端：响应式设计，支持移动浏览器

### 8.3 可扩展性
- 模块化设计，便于添加新功能
- 插件化的AI模型适配
- 灵活的配置系统

### 8.4 安全性
- API密钥加密存储
- 操作日志审计
- 敏感操作二次确认
- 暂不考虑复杂的权限控制（MVP阶段）

## 九、后续优化方向

1. **智能推荐**：根据用户行为推荐相关功能
2. **语音交互**：支持语音输入和输出
3. **多语言支持**：国际化
4. **个性化**：记住用户偏好和常用操作
5. **数据分析**：统计用户最常问的问题，优化知识库
6. **权限系统**：根据用户角色限制AI助手的能力范围
